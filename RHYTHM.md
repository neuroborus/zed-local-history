# Repository Rhythm

Chronological log of meaningful repo decisions. **Newest sections first:** add each new `## YYYY-MM-DD` block right below this paragraph, not at the end of the file.

## 2026-05-31

- Unix release bundles now ship `scripts/install.sh` for shell CLI installation: the installer copies `local-history`, `local-history-sidecar`, and `local-history-mcp` into `~/.local/bin` by default with `--prefix`, `--bin-dir`, and `--dry-run` overrides, prints a PATH hint when needed, and stays separate from the Zed extension's cached sidecar/MCP bootstrap; README documents the production CLI install path, Windows remains manual `.zip` extraction plus adding the chosen binary directory to `Path`.
- Fixed Zed Agent MCP launch path handling: extension-managed `context_server_command` now returns the resolved MCP executable path instead of a stable command name that Zed would reinterpret relative to the extension directory; manual fallback docs use Zed's top-level custom-server `command` / `args` / `env` settings shape and a distinct `local-history-dev` ID.
- Snapshot ID prefix contract unified in `local-history-core`: human CLI, MCP, and Markdown surfaces use the shared 12-character display prefix; restore/show/diff prefix lookup requires at least 6 characters and rejects shorter prefixes before scanning project databases.
- Contributor docs now have a short current-state entrypoint (`agents/CURRENT_STATUS.md`): `agents/README.md` and the root contributing section point coding agents there before long roadmap files, while stale `.gitignore` wording in planning docs is marked as future policy rather than current behavior.
- Watcher oversized-snapshot skips are now diagnosable: sidecar status JSON tracks `skipped_snapshot_count` and `last_skipped_snapshot`, the watcher log records skip metadata without file contents, and MCP status surfaces the same diagnostics in structured output and summary text.
- Human timestamp display now includes an explicit timezone suffix: CLI/MCP human output uses the local offset for the target timestamp and prints `UTC`, `+HH:MM`, or `+HH:MM:SS`; `init_local_offset_detection()` now actually enables `time` local-offset detection, while JSON, structured MCP fields, and generated Markdown keep canonical UTC behavior.
- Zed extension capabilities narrowed: removed wildcard `process:exec`; extension-side sidecar/MCP probes launch through stable command names with the resolved binary directory prepended to `PATH`, while extension-managed MCP server startup returns the resolved executable path to Zed. The manifest keeps only local-history binary names plus narrow `sh`/`where` MCP lookup helpers.
- Sidecar watcher daemon hardening: `watcher-status.json` is written via temp file + rename, `read_status` retries transient parse races, and `ensure_daemon` reuses a watcher when its PID is still alive even if the heartbeat is stale; status JSON now includes `process_alive`.
- Read-only storage inspection: `local-history-core` adds `open_read_only` / `open_for_snapshot_id_read_only` with SQLite read-only connections and `ensure_writable()` guards on mutating APIs; CLI/MCP list/view/diff/status use read-only open, restore/prune/snapshot stay on write paths, and sidecar `status`/`view_root` resolve layout without opening a writable store.
- Direct time filters validate RFC3339 before querying: `query_snapshots` returns `InvalidTimestamp` for bad bounds; CLI `resolve_time_filters` and MCP `resolve_time_filters` reject invalid `from`/`to`/`from_timestamp`/`to_timestamp` instead of silently returning empty results.
- MCP tool `content` for text-only hosts: `local_history_guide` returns the full `llms.txt` guide in `content[0].text`; `local_history_diff_snapshot` adds a fenced diff excerpt (20k-char cap with truncation notice) while keeping the full unified diff in `structuredContent.diff`.
- Pull-request CI now runs `cargo run -p xtask -- zed-ci` in a parallel `zed-extension` job (`wasm32-wasip2` toolchain) so extension changes are validated on every PR, not only locally. Release workflow bumps GitHub Actions to current majors (`actions/checkout@v6`, `actions/upload-artifact@v7`, `actions/download-artifact@v8`, `softprops/action-gh-release@v3`).
- Zed MCP PATH lookup on Windows: `resolve_lookup_binary_path` uses `zed::current_platform()` host `Os` (`where` on Windows, `command -v` via `sh` on Unix) instead of `cfg!(windows)`; failed PATH resolution falls back to cached/downloaded MCP while keeping version compatibility checks on PATH binaries.
- Unified diff hardening in `local-history-core`: line tokenization keeps trailing-newline state so `v1\n` vs `v1` is not reported as unchanged; common prefix/suffix trimming, a 1M-cell LCS cap, and replace-block fallback avoid quadratic memory on large CLI/MCP diffs.
- MCP recent-list presentation modes: `local_history_recent_snapshots` now defaults to `presentation=rich` (timestamp, path, id, one-line content preview in the text summary Zed shows agents), with `index` and `ids_only` for compact output; `llms.txt` and MCP `SERVER_INSTRUCTIONS` document the contract.
- Zed MCP startup hardening: extension resolves PATH/cached MCP binary directories before Agent Panel launch, avoids WASM `fs::metadata` probes on host binaries, adds manifest/spawn regression tests, and wires `cargo test` into `zed-ci` / `full-ci`; MCP adds stdio initialize smoke tests.
- Zed extension bootstrap fix for Agent Panel MCP: `extension.toml` now declares required `process:exec` and `download_file` capabilities; cached sidecar/MCP release binaries are resolved before `ProcessCommand` and context-server launch instead of assuming the host joins the extension work directory.
- Release workflow: replaced retired `macos-13` runner with `macos-15-intel` for macOS x86_64 builds. GitHub removed macOS 13 hosted runners in December 2025; jobs on `macos-13` wait indefinitely in queue.

The Zed extension now resolves `local-history-mcp` from `PATH` for local iteration, otherwise downloads and caches the matching fixed-name MCP release archive, checks MCP version compatibility before Agent Panel launch, and the release workflow now publishes MCP-only bootstrap archives alongside sidecar-only archives.

Stored snapshot IDs remain opaque, human tables now use 12-character prefixes, and CLI/sidecar/MCP restore paths resolve full IDs or unique prefixes with explicit ambiguity errors. ## 2026-05-03

- Closed the remaining non-manual MCP packaging gap. GitHub release bundles now stage and archive `local-history-mcp` alongside `local-history` and `local-history-sidecar`, so the README's packaged MCP binary path matches the actual release contract instead of only the development workspace layout.
- Added the first real MCP surface instead of leaving it as documentation-only architecture. The workspace now includes `crates/local-history-mcp`, a stdio JSON-RPC MCP server that implements `initialize`, `ping`, `tools/list`, and `tools/call`, with a first stable tool set for status, manual snapshot creation, recent snapshot listing, snapshot view, safety-first restore, and prune. Root docs now describe direct Zed `context_servers` usage, and the planning docs explicitly separate the implemented MCP slice from future MCP additions such as diff support and extension-managed registration.
- Promoted the root `README.md` from a thin project overview into the main self-contained user document. It no longer depends on `agents/` references for setup or usage context, and now documents the product surfaces, workflows, commands, storage model, privacy, troubleshooting, limitations, and an end-to-end example directly. `DEVELOPMENT_PLAN.md` also now ends with an explicit external validation plan for tagged releases, live Zed behavior, native installs, watcher realism, recovery safety, and README smoke testing.
- Stage 12 acceptance now has an explicit repository status instead of an all-or-nothing guess. The watcher/runtime tests now cover unchanged-content dedup behavior and atomic replace saves, and `agents/DEVELOPMENT_PLAN.md` now marks the MVP checklist items that are actually satisfied today while leaving the remaining real-world release validations unchecked.
- Stage 12 hardening now covers the missing recovery and workflow tests instead of only retention. `local-history-core` now has an explicit repeated-restore-chain regression test, `local-history-sidecar` now has an end-to-end workflow test that exercises scan/reconcile, recent snapshot lookup, restore, undo, Markdown render, and view rebuild, and `README.md` now documents privacy/storage behavior plus a first troubleshooting pass.
- Refined Stage 12 retention semantics so pruning cannot be bypassed by an ever-growing restore log. `prune` now preserves only the latest restore/undo chain needed for current undo behavior, explicitly deletes stale `restore_operations`, and releases older restore-pinned snapshots so storage budgets can actually converge again.
- Stage 12 retention work has started. `local-history-core` now exposes a default retention policy and a real `prune` path: snapshots larger than the current file-size cap are rejected, `prune` removes old or excess snapshots while preserving any restore-referenced snapshots, orphaned blobs are deleted safely, and the Markdown view is rebuilt after pruning. `local-history-cli` now exposes `local-history prune <project-root>` and includes the retention policy in `status`.
- Stage 11 release matrix coverage is wider now. The GitHub release workflow now includes native `aarch64-unknown-linux-gnu` and `aarch64-pc-windows-msvc` jobs using GitHub-hosted ARM64 runner labels, and the Zed extension bootstrap now has explicit asset mappings for Linux ARM64 and Windows ARM64 sidecar archives. The docs also now state explicitly that `x86_64-unknown-linux-musl` is still outside the extension bootstrap contract because the current platform mapping cannot distinguish Linux libc families.
- Stage 11 release hardening has started. The native sidecar now exposes a JSON `version` command, the Zed extension probes sidecar version compatibility before using a `PATH` or cached binary and falls back to the matching release asset when needed, and tagged GitHub releases now publish `SHA256SUMS.txt` alongside the platform archives and fixed-name sidecar bootstrap assets.

## 2026-05-02

- Stage 10 Zed command coverage now includes fixed 10-minute segment helpers instead of only hour-level wrappers. `local-history-sidecar` exposes `render-markdown current-segment` and `render-markdown segment-at --at <ISO-datetime>`, keeping segment-window normalization in the native layer, and the Zed extension now exposes matching `/local-history-current-segment` and `/local-history-segment <ISO-datetime>` slash commands over the same contract.
- Stage 10 sidecar bootstrap is now wired, not just slash-command plumbing. The Zed extension now resolves `local-history-sidecar` from `PATH` for dev usage, otherwise looks up the matching tagged GitHub release, downloads a platform-specific sidecar archive into the extension work directory, marks the binary executable when needed, and reuses that cached binary on later commands. `.github/workflows/release.yml` now also emits stable sidecar archive names for extension bootstrap while still publishing user-facing platform bundles to the GitHub Release.
- Stage 10 has its first real slice. `local-history-sidecar` now exposes Zed-facing wrapper commands for `view-root`, hour Markdown rendering, and `restore`, and the `editors/zed` extension now uses real slash-command handlers instead of bootstrap text. The extension resolves `local-history-sidecar` from `PATH`, starts the watcher with `ensure-daemon`, renders hour reports through the sidecar, and exposes generated Markdown paths back to the user. Automatic sidecar download/install is still intentionally deferred, so the current integration is path-first rather than release-bootstrap-first.
- Stage 9 Markdown views are now real. `local-history-core` can render an hour view or rebuild the full `view/` tree under external storage, with a root index, per-hour `README.md`, six fixed segment Markdown files, and exact snapshot Markdown pages that include restore commands and text previews. `local-history-cli` now exposes `render-markdown hour`, `render-markdown segment`, `view-root`, and `rebuild-markdown-view` as the filesystem-first browsing surface described in the plan.
- Stage 8 grouping is now real in `local-history-core` and `local-history-cli`. The core now exposes hour and segment history queries that group raw snapshots by affected file while preserving exact snapshot IDs and timestamps, and hour history is always split into six fixed 10-minute segments. The CLI now exposes `history hour` and `history segment` as text/JSON recovery queries. Grouping is intentionally additive: it sits on top of raw snapshots and never replaces precise restore targets.
- Stage 7 sidecar runtime is now real. `local-history-sidecar` gained a polling-based watcher that performs an initial ignored-file scan, caches current project state without snapshotting everything immediately, stores previous known contents on file change or deletion, and maintains a heartbeat-backed watcher status file. `ensure-daemon` now spawns a background watcher process and `status` reports watcher state from the sidecar logs directory. The implementation intentionally keeps scan/diff logic explicit and testable instead of hiding it inside OS-specific watcher callbacks.
- Documented an additional integration route around Zed's MCP server support without changing MVP scope. `GOALS.md`, `DEVELOPMENT_PLAN.md`, `README.md`, `editors/zed/README.md`, and `AGENTS.md` now treat a possible `local-history-mcp` crate as a thin adapter over `local-history-core`, usable either through direct `context_servers` settings or through extension-managed registration. The docs explicitly keep CLI and Markdown as the primary recovery surfaces and treat MCP as additive rather than as a replacement.
- Stage 6 CLI browsing is now in place. `local-history-core` exposes `SnapshotQuery` and `SnapshotPage` so pagination, file filters, and time-window filters live in one storage-level query path instead of being reimplemented in the CLI. `local-history-cli` now adds `list --page --page-size`, `--file`, `--from`, `--to`, `--hour`, `--json` for the current query commands, and a prompt-based `browse <project-root>` flow that can page through raw snapshots, preview one by number, and restore it only after explicit confirmation. The JSON contract is intentionally emitted only at the CLI boundary; `local-history-core` stays free of serialization concerns.
- `local-history-core` and `local-history-cli` now cover Stage 5 safety-first restore behavior instead of a destructive direct restore path. Every restore creates a `SnapshotKind::Safety` snapshot first, missing-file state is modeled explicitly on snapshots so undo can restore deletion as deletion instead of as an empty file, and `restore_operations` now record the safety snapshot, previous-file existence, and restored content hashes. The CLI now exposes `restore`, `restore --project-root <path> --recent <n>`, `safety-list`, `undo-restore`, and `restore-last-safety`, while `recent` intentionally shows raw snapshots only so safety bookkeeping does not disturb user-facing numbering.
- Bootstrapped `zed-local-history` as a Rust monorepo with `crates/local-history-core`, `crates/local-history-cli`, `crates/local-history-sidecar`, `xtask`, and a separate `editors/zed` package. The repository keeps project guidance under `agents/` instead of a single root `AGENTS.md`, and root docs now include `README.md`, `RHYTHM.md`, and `LICENSE`.
- Zed extension validation is intentionally split from the native workspace: the root repo stays on Rust `1.75.0` for conservative native compatibility, while `editors/zed` carries its own `rust-toolchain.toml` on `stable` with `wasm32-wasip2`. `xtask` now exposes `zed-fmt`, `zed-clippy`, `zed-check`, `zed-ci`, and `full-ci`, and explicitly clears inherited `RUSTUP_TOOLCHAIN` before nested `editors/zed` cargo runs so the extension package can honor its own toolchain file.
- Release automation now distinguishes generic Linux output from pinned Ubuntu 24.04 output in `.github/workflows/release.yml`: `ubuntu-latest` produces `local-history-x86_64-unknown-linux-gnu`, while a separate `ubuntu-24.04` job produces `local-history-x86_64-ubuntu-24.04`.
- `local-history-core` now has the first real Stage 3 storage layer: stable project identity is derived from normalized project root plus machine salt, storage layout stays external to the user repo under `projects/<project-id>/...`, default ignore policy is explicit, and a new `LocalHistoryStore` persists project metadata, content-addressed blobs, and snapshots through SQLite + compressed `.zst` blob files with round-trip and deduplication tests.
- `local-history-cli` now covers the first real Stage 4 recovery flow: `snapshot <project-root> --file <relative-path>` stores a manual raw snapshot, `recent <project-root>` lists newest snapshots from SQLite, `show <snapshot-id>` locates and previews a stored snapshot across projects, and `restore <snapshot-id>` performs the current basic restore path. Snapshot lookup across projects now uses `LocalHistoryStore::open_for_snapshot_id(...)`, while safety-first restore remains deferred to the next stage.
